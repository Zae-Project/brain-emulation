Documentation 1 · Project Overview & Goals

Welcome to the living documentation for our brain emulation visualizer. This reference replaces the old “lesson” flow with a guided handbook that covers every component of the project—from scientific data sources to the interactive interface that renders our spiking neural networks in real time. Each documentation page in this series can be read independently, and together they form a complete reference for contributors, students, and curious explorers.

Who is this for?

• Newcomers with no neuroscience or coding background who want a plain language introduction.
• Developers integrating new brain-region templates or neuron presets.
• Researchers and educators who need transparent sourcing for every dataset and assumption baked into the simulator.

Project Mission

The visualizer sits at the intersection of computational neuroscience, scientific storytelling, and practical tooling. Our mission is to demystify biologically grounded neural networks, provide a sandbox for hypothesis testing, and maintain an audit trail for the data that powers each simulation. The UI embraces cinematic clarity while keeping the underlying models transparent and reproducible.

High-Level Architecture

1. Data Layer

JSON templates in data/brain_region_maps/ describe clusters, neuron counts, and connectivity. Sources include the Allen Human Brain Atlas, EBRAINS Julich-Brain, and BrainGlobe Atlas API.

2. Template Registry

js/templates/registry.js loads builtin presets plus any manifest-discovered templates and exposes them to the UI via SNN_REGISTRY.

3. Simulation Core

js/app.js is the rendering + physics engine: it builds neuron instances, steps the spiking dynamics, draws glyphs, and feeds the inspector.

4. Documentation

The lessons/lesson*.html files (now “Documentation Chapters”) plus markdown notes in docs/ explain the interface, data pipelines, and research grounding.

Design Principles

  1. Transparency · Every visual element (neuron shape, template slider, inspector field) maps to a documented field in our schema. You should never guess where a number or glyph came from.
  2. Biological Plausibility · Default ratios (≈80% excitatory, 20% inhibitory) and connectivity probabilities mirror cortical literature (Beaulieu & Colonnier 1985, Ramaswamy et al. 2021).
  3. Modularity · You can drop new JSON templates, rebuild the manifest, and the UI will ingest them without code changes.
  4. Accessibility · The documentation aims for plain language first, technical detail second, so that non-experts can orient themselves quickly.
